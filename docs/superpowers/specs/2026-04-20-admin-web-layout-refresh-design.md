# 管理端固定侧栏与全站视觉提质设计稿

## 背景
当前管理端存在两个直接问题：
- 左侧菜单跟随页面整体滚动，桌面端导航不固定，长页面操作时定位不稳定。
- 整体视觉仍偏默认 Element Plus 后台，页面之间卡片、筛选区、表格区、标题层级缺少统一的设计语言。

本次设计目标是在不改动业务接口和路由结构的前提下，完成管理端壳层重构与全站视觉收口。

## 目标
- 桌面端左侧导航固定在视口内，右侧内容区独立滚动。
- 移动端改为顶部栏加抽屉导航，避免常驻侧栏挤压内容。
- 管理端统一为一套“编辑部式深色导航 + 轻雾白内容区”的视觉系统。
- 重点页面完成结构和视觉双升级，标准页面接入统一骨架与样式。
- 保持现有 API、路由、鉴权逻辑和主要业务流程不变。

## 非目标
- 不新增后端接口。
- 不重构现有数据流或状态管理。
- 不进行大规模共享组件拆分，只在明显重复且能降低维护成本时抽小型通用结构。
- 不把本次任务扩展为全新后台重写。

## 用户与使用场景
- 主要用户为桌面端管理员，长时间在视频管理、图片管理、上传中心、任务监控等页面切换。
- 次要用户为临时移动端查看管理员，需要可以进入页面、切换菜单、查看和执行基本操作，但不要求移动端达到桌面端同等效率。

## 设计方向
### 视觉关键词
- Data-Dense Dashboard
- Editorial Console
- Controlled Contrast
- Calm White Surface

### 色彩策略
- 主背景：冷白灰，降低当前全局偏粉底色。
- 导航区：深墨色为主，混合克制的酒红/品红层次。
- 强调色：统一用品红系，不再与蓝色并列作为主交互色。
- 辅助色：蓝色仅保留为图表或次级信息点缀，不参与主按钮和主导航竞争。

### 字体策略
- 保留 `Fira Sans` 作为正文与标题字体。
- 保留 `Fira Code` 用于数字型 KPI、ID、状态值、品牌标记。
- 强化页标题、区块标题、标签文字、辅助说明的层级差异。

### 质感与动效
- 内容区使用轻雾白卡片、清晰边框、较轻阴影。
- 交互反馈以颜色、描边、阴影变化为主，避免显著缩放带来的漂浮感。
- 保留轻量转场，并继续遵守 `prefers-reduced-motion`。

## 布局方案
### 桌面端
- 页面壳层高度固定为 `100vh`。
- 左侧导航为固定侧栏，始终处于视口内，不随内容滚动。
- 右侧区域拆分为顶部栏和主内容滚动区。
- 顶部栏只吸附在右侧内容区顶部，不覆盖整页。
- 主内容区成为主要滚动容器，避免 `body` 作为后台主滚动容器。

### 移动端
- 取消常驻左侧栏。
- 顶部栏增加菜单开关。
- 菜单内容进入 `drawer` 抽屉。
- 切换路由后自动关闭抽屉。
- 顶部栏始终可见，内容区独立滚动。

### 可访问性
- 增加“跳到主内容”链接，避免键盘用户每次穿过完整导航。
- 顶部固定区不遮挡首屏内容，内容区有明确的内边距补偿。
- 焦点态独立定义，确保导航项、按钮、筛选控件、抽屉开关可见。

## 统一页面骨架
所有后台页面统一采用四段式结构：
1. 页头：标题、说明、副操作区。
2. 概览区：统计卡或摘要信息。
3. 工具区：筛选、搜索、操作按钮。
4. 内容区：表格、图表、表单、上传面板或详情容器。

计划引入的通用类与语义层：
- `page-shell`
- `page-section`
- `section-head`
- `stats-grid`
- `toolbar-row`
- `content-card`
- `table-panel`

这些类由全局主题控制，页面只负责组织结构，减少每页重新定义布局节奏。

## 页面范围与改造策略
### 重点页面
#### Dashboard
- 作为新版视觉样板页。
- KPI 卡、趋势图卡片、页头和内容留白全部按新系统重做。
- 图表容器强化标题层次与背景稳定性。

#### VideoList
- 统一筛选区、表格区、详情弹窗的视觉层次。
- 表格外层转为更明确的结果面板，筛选和结果不再挤在同一视觉平面。
- 详情编辑表单同步新卡片与分组风格。

#### VideoUpload
- 上传区作为主任务面板突出显示。
- 表单区域按分组组织，进度区和结果区分别形成清晰的信息层。
- 与新增的“清空已选文件”交互保持一致，不再产生视觉歧义。

#### TaskMonitor
- 强化监控页的信息可读性。
- 进度条、状态、时间、错误信息列的层次更清晰。
- 工具区和表格区间距向新骨架对齐。

### 标准页面
- `ActorManage`
- `CollectionManage`
- `ImageManage`
- `ImageCollectionManage`
- `UserManage`
- `SystemSettings`

这些页面不做大幅业务结构改动，主要接入统一骨架、表格面板、筛选区和弹窗样式。

### 登录页
- 与后台主壳层保持同一设计语言。
- 保留独立场景，但色彩、卡片质感、标题层次、按钮风格统一到新版设计系统。

### 变更原则
- 脚本层只做必要的导航状态支持，例如移动端抽屉开关、路由切换关闭抽屉。
- 不主动改业务脚本和 API 调用逻辑。
- 优先通过模板结构类名和全局样式完成统一提质。
- 仅在明显有利于复用时，提取极小粒度的展示辅助结构。

## 关键交互细节
- 桌面端长页面滚动时，侧栏保持固定，顶部栏保持在右侧内容区顶部。
- 移动端抽屉打开后，用户可切换任一路由，切换成功后抽屉自动收起。
- 当前页面的菜单激活态更明确，包括颜色、背景、边界提示。
- 侧栏品牌区和顶部页头具有统一识别感，避免割裂。
- 表格 hover、行高、表头、分页、筛选控件间距统一。
- 所有可点击卡片、行、按钮、菜单项都有一致 hover/focus 表现。

## 验收标准
- 桌面端左侧菜单固定，不再随页面滚动。
- 右侧内容区独立滚动，顶部栏不遮挡内容。
- 移动端菜单使用抽屉，页面内容不被侧栏压缩。
- 管理端主要页面在视觉上属于同一套设计语言。
- 登录页与后台主界面风格一致但场景独立。
- 不引入新的业务回归。
- `cd admin-web && npm run build` 通过。

## 风险与控制
- 风险：全局主题调整会波及所有页面。
  - 控制：先以 `Layout.vue + theme.css` 收口共性，再对重点页做样板升级，标准页只做结构贴合。
- 风险：移动端抽屉导航与现有布局冲突。
  - 控制：将导航收拢在共享壳层，不在各页面内分散处理。
- 风险：视觉修改过多导致业务页可读性下降。
  - 控制：坚持“高密度后台”原则，优先信息清晰和操作效率，不做装饰性过强的视觉实验。
