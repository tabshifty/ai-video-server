# 家用部署机

本文档是 [[家用部署机]]（项目长期在线服务家人 TV / phone 客户端的那台机器，与 dev 模式 `dev-up.sh` 严格分家）的一次性安装步骤、日常运维命令和故障速查表。架构层面的取舍见 `docs/adr/0005-home-deployment-architecture.md` 与 `docs/adr/0006-migration-forward-compatibility.md`，本文档只写"怎么做"，不重复"为什么"。

适用前提：

- 部署机是 macOS（Apple Silicon 或 Intel 均可）。
- 部署机与开发机在同一 LAN，部署机有稳定的 hostname 或 LAN IP。
- 部署机外挂大容量盘已挂载在 `/Volumes/large`（用于 `STORAGE_ROOT`）。

## 一、前置依赖

部署机上需要：

```bash
# Go 工具链（与 go.mod 声明一致）
brew install go@1.22

# Node 工具链（admin-web 构建用）
brew install node

# FFmpeg 工具链（转码用）
# 家用部署默认使用 ffmpeg-full，保留 zscale/libzimg 等完整滤镜能力；
# 当前 DV→SDR 可信输出方案已撤回，不能把 zscale 可用等同于 DV 安全转 SDR。
brew install ffmpeg-full

# Docker Desktop（跑 Postgres / Redis 容器）
# 从官网下载或 brew install --cask docker

# git（macOS 自带通常够新；如需更新版本：brew install git）
```

`ffmpeg-full` 是 keg-only，不会自动替换 `/opt/homebrew/bin/ffmpeg`。本项目通过 launchd 的 PATH 让 server / worker 优先使用 `/opt/homebrew/opt/ffmpeg-full/bin/ffmpeg`，不要用 `brew link ffmpeg-full` 改全局链接。

确认外挂盘可写：

```bash
ls -la /Volumes/large && mkdir -p /Volumes/large/ai-video-server/storage
```

启用 SSH 远程登录（开发机 push 走 ssh，否则连不上）：

```bash
sudo systemsetup -setremotelogin on
# 或在系统设置 → 共享 → Remote Login 打开
# 验证：开发机 `ssh chee@<部署机 IP>` 能登入
```

如果部署机 LAN IP 经常变，可考虑用 mDNS 名（`*.local`，前提是路由器支持）或在路由器侧给它分配静态 IP。

## 二、一次性安装

以下步骤在部署机上跑一次。`${USER}` 自动展开为当前用户名（**带大括号，不要写 `$USER:staff` 否则 zsh 会把它当 history substitution，把 `chee` 改成 `cheeaff`**）。

### 1. 初始化部署目录树

```bash
mkdir -p ~/deploy/ai-video-server/{repo.git,work,binaries,current/admin-web-dist}
mkdir -p ~/Library/Logs/ai-video-server

# 初始化 bare repo
cd ~/deploy/ai-video-server/repo.git
git init --bare

# git 2.28+ 的 init.defaultBranch 默认是 main，但本项目走 master。
# 把 HEAD 切到 master，避免 push --mirror 顺手把 main 镜像到 GitHub。
git symbolic-ref HEAD refs/heads/master
```

部署目录最终长这样：

```
~/deploy/ai-video-server/
├── repo.git/                    bare repo，push 目标
├── work/                        checkout（hook 内 reset --hard）
├── binaries/                    历史 Go 二进制（保留 3 个，供 rollback）
├── current/
│   ├── video-server              稳定运行入口文件
│   └── admin-web-dist/          已构建前端
└── (storage 不在这里，见 .env)
```

### 2. 准备部署机的 `.env`

部署机的 `.env` 必须**全用绝对路径**（见 [[家用部署机绝对路径契约]]）。建议位置 `~/deploy/ai-video-server/.env`。

⚠️ **`.env` 文件首字节不能有 UTF-8 BOM**（`ef bb bf`）：godotenv 会把 BOM 当 var-name 前缀，导致首条变量（甚至全部解析）被破坏，server 启动报 `POSTGRES_DSN is required` 但你"明明写了"。dev-up.sh 不读 godotenv 所以历史上看不到这条坑。开发机如果用 VSCode / Sublime 默认 UTF-8 with BOM 编辑过 `.env`，复制到部署机前必须先剥 BOM：

```bash
# 检查
head -1 .env | xxd | head -1     # 期望开头是 "41 50 50"（APP），不是 "ef bb bf"
# 剥 BOM
sed -i.bom -e '1s/^\xef\xbb\xbf//' .env && rm -f .env.bom
```

下面是一个示例（占位值需替换为真实凭据）：

```bash
cat > ~/deploy/ai-video-server/.env <<'EOF'
APP_MODE=server
HTTP_ADDR=:8080

# 数据层（Docker 容器，详见第 3 步）
POSTGRES_DSN=postgres://video:video@127.0.0.1:5432/video_server?sslmode=disable
REDIS_ADDR=127.0.0.1:6379
REDIS_PASSWORD=

# 媒体路径——必须绝对，指向外挂盘
STORAGE_ROOT=/Volumes/large/ai-video-server/storage
POSTER_STORAGE_PATH=/Volumes/large/ai-video-server/storage/posters

# 上传暂存目录——必须留在本机磁盘或工作目录，不要放到 /Volumes/large
UPLOAD_TEMP_DIR=./tmp/uploads

# launchd 二进制稳定签名身份——供 post-receive hook / rollback 重新 codesign
# 必须是 Apple-issued 的 Developer ID Application / Apple Development 证书；
# 仅用 OpenSSL 自签证书不够，`codesign` 不会把它当成长期开服身份。
CODESIGN_IDENTITY="Developer ID Application: <你的名字> (<TEAMID>)"
# 固定 codesign identifier，避免版本化文件名影响 TCC 主体识别。
CODESIGN_IDENTIFIER=com.chee.videos.server

# 可选：若 identity 放在独立 keychain，可在这里指定路径并配套密码
#（脚本会在签名前尝试解锁它）
CODESIGN_KEYCHAIN="$HOME/Library/Keychains/ai-video-server.keychain-db"
# CODESIGN_KEYCHAIN_PASSWORD="<keychain 密码>"
# 或改用只对当前用户可读的密码文件，避免把 keychain 密码明文写进 .env。
# CODESIGN_KEYCHAIN_PASSWORD_FILE="$HOME/.codesign/ai-video-server.keychain.pass"

# admin-web dist 路径——必须绝对，指向 current/ 下的产物
# （依赖 internal/handlers/router.go 支持 ADMIN_WEB_DIST_PATH；
#  若代码尚未支持此 env，请先完成对应 task 再启动 server）
ADMIN_WEB_DIST_PATH=/Users/<你的用户名>/deploy/ai-video-server/current/admin-web-dist

# 翻译服务（按需）
TRANSLATION_API_URL=http://127.0.0.1:8000/v1

# 其它从 .env.example 抄过来按家庭网络情况调整
EOF
```

把 `<你的用户名>` 替换成真实 macOS 用户名。其它键参考 `.env.example`。

如果开发机要复用部署机翻译服务（见 [[远程支撑层开发模式]]），部署机上的 OpenAI-compatible 翻译端点必须能被开发机访问。最小改法是让 `llama-server` 监听 LAN：

```bash
llama-server ... --host 0.0.0.0 --port 8000
```

然后开发机 `.env.remote-local` 使用：

```bash
TRANSLATION_API_URL=http://<部署机 IP>:8000/v1
```

如果不希望在 LAN 暴露 8000 端口，也可以只让部署机继续监听 `127.0.0.1:8000`，开发机通过 SSH tunnel 暴露本地端口：

```bash
ssh -N -L 18000:127.0.0.1:8000 chee@<部署机 IP>
```

再把开发机 `.env.remote-local` 写成：

```bash
TRANSLATION_API_URL=http://127.0.0.1:18000/v1
```

### 3. 启动 Postgres / Redis 容器（独立生命周期）

数据层由 docker compose 单独管，不跟随代码 push 重启（见 [[家用部署机数据层独立生命周期]]）。建议在 `~/deploy/ai-video-server/` 放一份 `docker-compose.yml`：

```yaml
services:
  postgres:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_USER: video
      POSTGRES_PASSWORD: video
      POSTGRES_DB: video_server
    ports: ["127.0.0.1:5432:5432"]
    volumes:
      - /Volumes/large/ai-video-server/pgdata:/var/lib/postgresql/data
  redis:
    image: redis:7
    restart: unless-stopped
    ports: ["127.0.0.1:6379:6379"]
    volumes:
      - /Volumes/large/ai-video-server/redisdata:/data
```

```bash
cd ~/deploy/ai-video-server
docker compose up -d
```

`restart: unless-stopped` 保证 Mac 重启后容器自动起。

### 4. 安装 launchd plist（supervisor）

为 Go server 与 worker 各装一份。下面以 server 为例（worker 同结构，把 `-mode server` 改成 `-mode worker`，label 改成 `com.aivideo.worker`）：

```bash
cat > ~/Library/LaunchAgents/com.aivideo.server.plist <<EOF
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key><string>com.aivideo.server</string>
  <key>ProgramArguments</key>
  <array>
    <string>$HOME/deploy/ai-video-server/current/video-server</string>
    <string>-mode</string><string>server</string>
  </array>
  <key>WorkingDirectory</key>
  <string>$HOME/deploy/ai-video-server/current</string>
  <key>EnvironmentVariables</key>
  <dict>
    <key>PATH</key>
    <string>/opt/homebrew/opt/ffmpeg-full/bin:/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin</string>
    <key>ENV_FILE</key>
    <string>$HOME/deploy/ai-video-server/.env</string>
  </dict>
  <key>KeepAlive</key><true/>
  <key>RunAtLoad</key><true/>
  <key>StandardOutPath</key>
  <string>$HOME/Library/Logs/ai-video-server/server.log</string>
  <key>StandardErrorPath</key>
  <string>$HOME/Library/Logs/ai-video-server/server.log</string>
</dict>
</plist>
EOF

# worker 同理
cp ~/Library/LaunchAgents/com.aivideo.server.plist \
   ~/Library/LaunchAgents/com.aivideo.worker.plist
sed -i '' \
  -e 's/com.aivideo.server/com.aivideo.worker/g' \
  -e 's|<string>-mode</string><string>server</string>|<string>-mode</string><string>worker</string>|' \
  -e 's|server.log|worker.log|g' \
  ~/Library/LaunchAgents/com.aivideo.worker.plist

# 此时还不能 load，因为 current/video-server 还不存在；先跳到第 6 步推第一次 push 后再回来 load
# launchd 永远只指向这个稳定入口，不要把 `binaries/video-server-<sha>.bin`
# 直接写进 plist，否则每次重编译都可能被 macOS 当成新的访问主体。
```

安装或调整 plist 后，确认 launchd 环境能选中 `ffmpeg-full` 且具备 `zscale`：

```bash
PATH="/opt/homebrew/opt/ffmpeg-full/bin:/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin" \
  ffmpeg -hide_banner -filters | grep -E '(^|[[:space:]])(zscale|tonemap)([[:space:]]|$)'
```

> 外挂盘权限提醒：server / worker 由 launchd 启动，不能只验证 Terminal 或 SSH 手动命令能读 `/Volumes/large`。macOS TCC 会按进程代码身份判定外置卷/完整磁盘访问；如果 launchd 版 server 访问封面或视频卡在 `open`，但手动 `dd if=/Volumes/large/... bs=1 count=1` 很快，优先检查 [[家用部署机媒体访问权限契约]] 和 [[家用部署机稳定签名契约]]。ad-hoc 二进制只适合临时手工救火；长期部署必须对同一份稳定签名身份重新 codesign，首次切换后再对新二进制重新授权一次。

> 授权补充：`security add-trusted-cert` / `trust-settings-import` 这类 trust settings 变更，macOS 不允许在纯 SSH 无交互环境里替用户完成。也就是说，签名身份可以由 hook 自动复用，但“第一次把这份身份授信给系统”需要在部署机本地的 GUI / 管理员会话里做一次；`security find-identity -v -p codesigning` 必须先能看到有效 identity，后面的 push 才有意义。

#### 只做服务器时的权限收口方案

如果这台机器只承担服务器职责，不想反复碰 TCC，优先级是：

1. 能迁就迁：把 `STORAGE_ROOT`、`POSTER_STORAGE_PATH`、`UPLOAD_TEMP_DIR` 这类运行期目录迁到内置盘或本机盘，直接消掉外盘依赖。
2. 不能迁再授：如果媒体必须继续放在 `/Volumes/large`，保持 `server` / `worker` 用同一份稳定签名身份，并在部署机本地 GUI 里给这两个 launchd binary 授予 `Full Disk Access` 或外部卷访问。
3. 不要找 SSH 绕过：纯 SSH 没有“永久解除 TCC 限制”的办法，launchd 进程要么拿到本地授信，要么就继续按外盘受限处理。

若稳定签名已经生效，但每次 push 后仍为外盘访问反复弹授权窗，下一档方案是收紧为 [[家用部署机固定执行路径契约]]：launchd 永远执行固定真实文件（例如 `~/deploy/ai-video-server/current/video-server.bin`，server / worker 继续通过 `-mode` 区分），hook 仍保留 `binaries/video-server-<sha>.bin` 作为回滚备份，但发布时把目标二进制替换到固定路径并用同一 identity 重新 codesign。部署机 GUI 只给这个固定路径授一次 `Full Disk Access` / 外置卷访问，目标是不再每个 commit 重新点授权。

### 5. 安装 newsyslog 日志切割规则

```bash
sudo tee /etc/newsyslog.d/ai-video-server.conf >/dev/null <<EOF
# logfilename                                            [owner:group]  mode count size when  flags
${HOME}/Library/Logs/ai-video-server/server.log          ${USER}:staff    644  5     10240 *    GJ
${HOME}/Library/Logs/ai-video-server/worker.log          ${USER}:staff    644  5     10240 *    GJ
${HOME}/Library/Logs/ai-video-server/deploy.log          ${USER}:staff    644  5     10240 *    GJ
EOF
```

含义：每个文件 10240 KB（10 MB）切一次，保留 5 个 gz 归档，`GJ` 表示 gzip 压缩 + 在 launchd 触发时点切割。**注意必须用 `${USER}` 不是 `$USER`**：zsh 把 `$USER:staff` 当作 history-style substitution，会把 `chee` 改成 `cheeaff` 之类的乱码。bash 不会，但仓库默认 shell 是 zsh，写 `${...}` 是统一安全写法。

### 6. 写 post-receive hook（核心）

把以下内容写入 `~/deploy/ai-video-server/repo.git/hooks/post-receive`：

```bash
#!/usr/bin/env bash
set -Eeuo pipefail

DEPLOY_ROOT="$HOME/deploy/ai-video-server"
WORK="$DEPLOY_ROOT/work"
BIN_DIR="$DEPLOY_ROOT/binaries"
CURRENT="$DEPLOY_ROOT/current"
LOG="$HOME/Library/Logs/ai-video-server/deploy.log"
LOCKDIR="$DEPLOY_ROOT/deploy.lock.d"
GITHUB_MIRROR="https://github.com/tabshifty/ai-video-server.git"

# 把 brew / node 加进 PATH，hook 在 launchd 或 ssh 触发时 PATH 可能很瘦
export PATH="/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin"

log() { printf '[%s] %s\n' "$(date '+%F %T')" "$*" | tee -a "$LOG"; }

# 串行化 push（[[家用部署机]] Q7c）。macOS 没有 flock，用 mkdir 做原子锁。
acquire_lock() {
  local tries=0
  until mkdir "$LOCKDIR" 2>/dev/null; do
    tries=$((tries + 1))
    if [[ $tries -gt 600 ]]; then
      log "等锁 $LOCKDIR 超过 10 分钟，放弃"
      exit 1
    fi
    sleep 1
  done
  trap 'rmdir "$LOCKDIR" 2>/dev/null || true' EXIT
}
acquire_lock

while read oldsha newsha refname; do
  [[ "$refname" == "refs/heads/master" ]] || { log "skip non-master ref: $refname"; continue; }
  log "==== deploy $oldsha → $newsha ===="

  # 同步 GitHub 镜像（best-effort，失败不阻塞部署）
  git --git-dir="$DEPLOY_ROOT/repo.git" push --mirror "$GITHUB_MIRROR" 2>>"$LOG" || log "github mirror push failed (non-fatal)"

  # checkout 到 work
  git --git-dir="$DEPLOY_ROOT/repo.git" --work-tree="$WORK" reset --hard "$newsha"
  git --git-dir="$DEPLOY_ROOT/repo.git" --work-tree="$WORK" clean -fdx -e storage

  # 分桶判定（[[家用部署机 push 分桶]]）
  CHANGED=$(git -c core.quotepath=false --git-dir="$DEPLOY_ROOT/repo.git" diff --name-only "$oldsha" "$newsha")
  RESTART_GO=0
  REBUILD_FRONTEND=0
  while IFS= read -r f; do
    [[ -z "$f" ]] && continue
    case "$f" in
      main.go|main_test.go|cmd/*|internal/*|pkg/*|go.mod|go.sum|migrations/*)
        RESTART_GO=1 ;;
      admin-web/dist/*)
        ;; # 显式跳过：dist 不该入库，若意外入库忽略
      admin-web/*)
        REBUILD_FRONTEND=1 ;;
      android-app/*|android-tv-app/*|docs/*|tasks/*|plan.md|plan.archive.md|CONTEXT.md|AGENTS.md|README.md|.gitignore|.env.example|scripts/*|skills-lock.json|.agents/*|.codex/*|references/*|.run/*)
        ;; # 显式跳过
      *)
        log "未归桶路径 $f —— 视作跳过（如需触发，请扩展 hook 的 case 分支）"
        ;;
    esac
  done <<< "$CHANGED"

  log "RESTART_GO=$RESTART_GO REBUILD_FRONTEND=$REBUILD_FRONTEND"

  # 前端 build
  if [[ "$REBUILD_FRONTEND" == "1" ]]; then
    log "==> npm ci && npm run build"
    (cd "$WORK/admin-web" && npm ci && npm run build) || { log "npm build FAILED, keeping old dist"; continue; }
    rm -rf "$CURRENT/admin-web-dist.new"
    cp -R "$WORK/admin-web/dist" "$CURRENT/admin-web-dist.new"
    mkdir -p "$CURRENT/admin-web-dist/assets"
    if [[ -d "$CURRENT/admin-web-dist.new/assets" ]]; then
      rsync -a "$CURRENT/admin-web-dist.new/assets/" "$CURRENT/admin-web-dist/assets/" || {
        log "frontend assets sync FAILED, keeping old dist"
        rm -rf "$CURRENT/admin-web-dist.new"
        continue
      }
    fi
    rsync -a --exclude assets "$CURRENT/admin-web-dist.new/" "$CURRENT/admin-web-dist/" || {
      log "frontend root sync FAILED, keeping old dist"
      rm -rf "$CURRENT/admin-web-dist.new"
      continue
    }
    find "$CURRENT/admin-web-dist/assets" -type f -mtime +7 -delete 2>>"$LOG" || log "old frontend asset prune failed (non-fatal)"
    rm -rf "$CURRENT/admin-web-dist.new"
    log "frontend updated"
  fi

  # Go build + migrate + restart
  if [[ "$RESTART_GO" == "1" ]]; then
    short="${newsha:0:10}"
    BIN_NEW="$BIN_DIR/video-server-$short.bin"
    log "==> go build → $BIN_NEW"
    (cd "$WORK" && go build -o "$BIN_NEW" .) || { log "go build FAILED, keeping old binary"; continue; }

    log "==> codesign → $BIN_NEW"
    CODESIGN_ENV_FILE="$DEPLOY_ROOT/.env" "$WORK/scripts/sign-launchd-binary.sh" "$BIN_NEW" || { log "codesign FAILED, keeping old binary"; continue; }

    log "==> apply migrations"
    POSTGRES_CONTAINER=video_server_postgres "$WORK/scripts/migrate-apply.sh" 2>>"$LOG" || {
      log "migrate FAILED, keeping old binary"
      continue
    }

    # 稳定运行入口：launchd 永远只看 `current/video-server`，
    # 版本化二进制只留在 binaries/ 里做 rollback 备份。
    cp "$BIN_NEW" "$CURRENT/video-server.new"
    chmod 755 "$CURRENT/video-server.new"
    mv -f "$CURRENT/video-server.new" "$CURRENT/video-server"

    log "==> launchctl kickstart"
    launchctl kickstart -k "gui/$(id -u)/com.aivideo.server" || true
    launchctl kickstart -k "gui/$(id -u)/com.aivideo.worker" || true

    # 探活（不做自动 rollback，只回报）
    sleep 2
    if curl -fsS --max-time 5 --retry 10 --retry-delay 1 http://127.0.0.1:8080/healthz >/dev/null; then
      log "/healthz OK — deploy succeeded"
    else
      log "/healthz FAILED — service may be down. 检查 ~/Library/Logs/ai-video-server/server.log"
    fi

    # 保留最近 3 个二进制
    ls -t "$BIN_DIR"/video-server-*.bin 2>/dev/null | tail -n +4 | xargs -r rm -f
  fi

  log "==== deploy $newsha done ===="
done
```

```bash
chmod +x ~/deploy/ai-video-server/repo.git/hooks/post-receive
```

### 7. 部署机最后开服

第一次推 push 之前先在部署机加载 launchd plist：

```bash
launchctl bootstrap "gui/$(id -u)" ~/Library/LaunchAgents/com.aivideo.server.plist
launchctl bootstrap "gui/$(id -u)" ~/Library/LaunchAgents/com.aivideo.worker.plist

# 验证
launchctl print "gui/$(id -u)/com.aivideo.server" | grep -E 'state|pid'
curl -s http://127.0.0.1:8080/healthz
```

`KeepAlive=true` 会保证服务长期在线，重启 Mac 也自动起。

## 三、开发机一次性配置

完整把"开发机 push → 部署机自动构建并重启"这条链路打通。

### 1. 生成 SSH 公私钥（如已有可跳过）

在**开发机**上：

```bash
ls ~/.ssh/id_*.pub 2>/dev/null || ssh-keygen -t ed25519 -N '' -f ~/.ssh/id_ed25519
```

### 2. 推公钥到部署机做免密登录

```bash
# 这一步会要部署机 chee 用户密码 —— 这是最后一次输密码
ssh-copy-id chee@<部署机 LAN IP>

# 例如：
# ssh-copy-id chee@192.168.1.24
```

`ssh-copy-id` 会把开发机 `~/.ssh/id_ed25519.pub` 追加到部署机 `~/.ssh/authorized_keys`。部署机 `~/.ssh/` 权限 700、`authorized_keys` 权限 600 才会被 sshd 接受；初次安装时部署机一次性安装文档已经处理好。

### 3. 验证免密

```bash
ssh chee@<部署机 LAN IP> 'hostname && date'
```

期望输出 `CheedeMac-mini.local` + 当前时间，**不应**再问密码。若仍要密码，回到部署机检查 `~/.ssh/authorized_keys` 是否含开发机的公钥行（应有 `ssh-ed25519 AAAA... chee@<开发机名>` 一行）以及 `~/.ssh/` 权限。

### 4. 加 deploy git remote

```bash
# 在开发机的本项目 checkout 目录里
git remote add deploy chee@<部署机 LAN IP>:/Users/chee/deploy/ai-video-server/repo.git

# 验证：列两个 remote
git remote -v
# 应该看到：
#   deploy  chee@192.168.1.24:/Users/chee/deploy/ai-video-server/repo.git
#   origin  https://github.com/tabshifty/ai-video-server.git
```

### 5. 首次 push 触发 hook

```bash
git push deploy master
```

push 终端会**实时回流** hook 的全部输出（build / migrate / kickstart / `/healthz`）。如果是首次推一个含 Go 改动的 commit，看到的流程大致是：

```
remote: [...] ==== deploy <old> → <new> ====
remote: [...] RESTART_GO=1 REBUILD_FRONTEND=1
remote: [...] ==> npm ci && npm run build
remote: [...] ==> go build → .../binaries/video-server-<short>.bin
remote: [...] ==> apply migrations
remote: [...] ==> launchctl kickstart
remote: [...] /healthz OK — deploy succeeded
remote: [...] ==== deploy <new> done ====
```

任一阶段失败 → fail-open，旧进程继续跑，错误堆栈在 push 终端 stderr 里给你直接看（不用 ssh 过去）。

### 6. 之后日常

```bash
# 在开发机正常工作
# 改代码 → 看 admin SPA 改动 → npm run dev / go test ...
# 完事了：
git add -A && git commit -m '...'
git push deploy master   # 触发部署
git push origin master   # 通常不必，hook 已经 --mirror 同步到 GitHub
```

`git push origin` 唯一必要的场景是 hook 同步 GitHub 失败时（部署机看不到外网），手动补一次。否则只 push deploy 就够。

## 四、日常运维

### 看日志

```bash
# 部署历史与每次 deploy 的执行细节
tail -f ~/Library/Logs/ai-video-server/deploy.log

# Go server 实时日志
tail -f ~/Library/Logs/ai-video-server/server.log

# Worker 实时日志
tail -f ~/Library/Logs/ai-video-server/worker.log
```

### 手动重启

```bash
launchctl kickstart -k "gui/$(id -u)/com.aivideo.server"
launchctl kickstart -k "gui/$(id -u)/com.aivideo.worker"
```

### 看服务状态

```bash
launchctl print "gui/$(id -u)/com.aivideo.server" | grep -E 'state|last exit|pid'
```

### 手动 rollback

```bash
~/deploy/ai-video-server/work/scripts/rollback.sh <commit-sha>
```

具体行为见 `scripts/rollback.sh` 自身。要点：

- 只把选中的历史二进制复制回 `current/video-server` 再重启 launchd，**不自动跑 migration down**。
- 依赖 [[migration 前向兼容契约]]（ADR-0006）—— 旧 binary 必须能在新 schema 上跑。
- 如果回退跨越大量 commit 跨度，本地可能已无对应二进制（只保留 3 个），需要先 `git checkout <sha> && go build` 重建。

## 五、故障速查

| 现象 | 排查 |
|---|---|
| 每次重新编译后都被 macOS 再次拦截外盘权限 | 确认 launchd 只指向 `current/video-server`，不要直接启动 `binaries/video-server-<sha>.bin`；如果已经给 Terminal / IDE 授权，就始终复用同一个宿主来触发部署，不要换启动器 |
| push 报错 / hook 输出有 FAILED | 看 push 终端 stderr 完整内容，或 `deploy.log` 最新一段；构建 / 迁移失败旧版本仍在跑（fail-open） |
| `/healthz` 探活失败但 hook 没崩 | `tail -100 ~/Library/Logs/ai-video-server/server.log` 找崩溃原因；常见：env 缺失、Postgres 不可达、端口被占 |
| TV 客户端连不上 | 检查 server 进程：`launchctl print "gui/$(id -u)/com.aivideo.server"`；端口：`lsof -i :8080`；防火墙：系统设置 → 网络 → 防火墙 |
| 上传/转码任务卡住 | 重启 worker：`launchctl kickstart -k "gui/$(id -u)/com.aivideo.worker"`；Asynq 会重派 [[家用部署机重启可恢复边界]] |
| DV 视频转码失败并报 `No such filter: 'zscale'` | 说明 worker 实际执行的 ffmpeg 缺少 `zscale`。先查 `launchctl print "gui/$(id -u)/com.aivideo.worker"` 里的 PATH 是否以 `/opt/homebrew/opt/ffmpeg-full/bin` 开头，再用同一 PATH 跑 `ffmpeg -hide_banner -filters | grep zscale`。单独 `brew install zimg` 不会让普通 `/opt/homebrew/bin/ffmpeg` 自动获得 `zscale`。 |
| 管理端列表能打开但图片 503、视频 source pending | 先区分三件事：`/api/v1/admin/videos` 是否 200、部署机 shell 是否能 `dd if=/Volumes/large/... bs=1 count=1`、launchd server 采样是否卡在 `open`。若 shell 能读但 launchd 卡住，按 [[家用部署机媒体访问权限契约]] 和 [[家用部署机稳定签名契约]] 排查 TCC / 签名身份，不要只重启或改数据库路径 |
| migration 报错 | 看 `deploy.log` 中的 SQL 错误；旧 schema 仍兼容旧 binary，无须紧急；改 migration 后再 push |
| Postgres / Redis 容器挂 | `docker ps`；`docker compose -f ~/deploy/ai-video-server/docker-compose.yml restart` |
| 磁盘满（外挂盘） | `df -h /Volumes/large`；`storage/` 下的孤儿临时文件清理是独立 task，参见 [[家用部署机重启可恢复边界]] |

## 六、弃用与清理

如需停用 [[家用部署机]] 模式：

```bash
launchctl bootout "gui/$(id -u)" ~/Library/LaunchAgents/com.aivideo.server.plist
launchctl bootout "gui/$(id -u)" ~/Library/LaunchAgents/com.aivideo.worker.plist
rm ~/Library/LaunchAgents/com.aivideo.{server,worker}.plist
docker compose -f ~/deploy/ai-video-server/docker-compose.yml down
sudo rm /etc/newsyslog.d/ai-video-server.conf
# 部署目录与媒体文件保留在原位，按需手动清理
```

GitHub 端不受影响（一直保持镜像）。
