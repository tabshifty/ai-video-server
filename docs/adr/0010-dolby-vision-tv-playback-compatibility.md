# TV 端杜比视界播放兼容策略以不异色优先

TV 端面对 Dolby Vision 风险源时，首要目标是不把可能异色的视频交给不可信链路播放。当前已经可验证的是后端把 source/output 的播放兼容探测结果持久化到 `playback_compat` metadata，TV 端据此做保守阻断；这不等于项目已经具备 Dolby Vision 安全放行能力。因此本决策确认：已明确标记为 Dolby Vision 风险源的视频，只有在当前仍可访问的播放源自身仍是 Dolby Vision、设备显示能力与目标播放链路能力都可确认支持，并且走专用系统播放链路时，才有资格进入 Dolby Vision 安全放行；现有 LibVLC 普通长视频链路不承担 Dolby Vision 安全播放。

这个策略刻意保留了一个历史兼容例外：`playback_compat` 整体缺失或协议版本过旧的存量视频，继续按既有 LibVLC 普通链路播放。它只是为了避免历史数据在升级后被大面积阻断，不代表这些视频已经完成 Dolby Vision 安全判断，也不能进入专用系统播放链路。一旦 metadata 已存在并明确标记 Dolby Vision 风险，或 metadata 已存在但探测失败、source/output 判定不完整，就不再适用老数据放行，而是进入阻断或“播放兼容信息不完整”的提示路径。

播放入口不做实时 ffprobe、不触发后台补探测、不隐式重新转码，也不提供“忽略风险继续播放”的旁路。重试只表示重新拉取详情 metadata、重新执行本地能力判断和播放决策；真正的补探测或 metadata 修复应放到后端任务或管理端修复入口另行设计。这样会牺牲一部分短期可播性，但换来的是清晰的安全边界、稳定的播放入口延迟，以及避免把已知可能异色的链路伪装成安全回退。

## 考虑过的替代方案

- **所有 metadata 缺失或不完整的视频一律阻断**：安全边界最干净，但会让大量存量视频在升级后突然不可播，家庭使用体验不可接受。
- **metadata 不完整时乐观猜测可播放**：短期可播性最好，但会把文件名、类型或历史经验当成安全依据，和“不异色优先”的目标冲突。
- **在播放页实时补探测或触发后台修复**：看似能自动恢复，但会把播放入口绑定到外盘响应、TCC 授权、ffprobe 耗时和后端任务状态，容易让播放页不可预测。
- **让 LibVLC 普通链路承担 Dolby Vision 安全放行**：实现最省事，但当前没有证据证明它能安全输出对应 Dolby Vision profile；在已有异色风险背景下，这会把风险重新引回主播放链路。
