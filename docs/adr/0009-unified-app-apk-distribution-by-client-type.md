# 手机端与 TV 端共用安装包分发模型，并按客户端类型分轨

安装包分发能力在 TV 端落地后，Android 手机端也需要进入同一套管理体系；但手机端与 TV 端的安装包形态并不完全一致：TV 端继续保留按 ABI 拆分的 `armeabi-v7a` / `arm64-v8a` 双包分发，而手机端当前工程事实是 `applicationId = com.chee.videos`，首期不做 ABI 拆分，默认一条发布记录只对应一个 APK。这个决策因此一次性确定：**手机端与 TV 端共用同一套“发布记录 + 安装包产物”分发模型，但按客户端类型分轨**，至少区分 `android_phone` 与 `android_tv`。

之所以不做“两套完全独立的手机端/TV 端分发系统”，是因为两边共享的上位语义已经足够多：都需要管理员上传本地 Release APK、都需要发布/下线/恢复、都需要最近三版回退窗口、都需要家庭成员登录后手动下载安装、都需要推荐版本自动指向当前可见范围里的最高 `versionCode`。真正的分叉只应保留在**安装包形态**与**家庭页展示维度**上：TV 端是一条发布记录下挂多个 ABI 安装包，并显式展示已上传/缺失 ABI；手机端是一条发布记录对应一个 APK，不再引入 ABI 完整性维度。

因此管理端也不应顺手演化成两套平行页面。Admin Web 的方向是一个上位“安装包管理”工具页，在页内按客户端类型切换 `android_phone` / `android_tv`，把“共模型、分轨”的结构直接表达出来；如果拆成两个完全独立页面，会把本来是同一套领域模型的问题退化成两套 UI / API / 测试的重复维护。相反，家庭成员侧的轻量下载页需要分开：手机端与 TV 端虽然共用登录体系和最近三版/推荐位等规则，但终端用户看到的安装决策信息不同，TV 端要看 ABI，手机端要看单包版本说明、发布时间和文件大小；若混在同一家庭下载页里，误下错误客户端安装包的风险过高。

首期还一次性收紧几个约束，避免“共模型”带来歧义：系统必须强校验 `client_type` 与 `packageName` 的对应关系，而不是只靠管理员记忆；当前固定为 `android_tv -> com.chee.videos.tv`、`android_phone -> com.chee.videos`。在同一 `client_type + packageName` 分发轨内，`versionCode` 必须唯一，`versionName` 只用于展示和校验辅助；若命中同 `versionCode` 但 `versionName` 不同，视为非法上传并拒绝归档。手机端与 TV 端都继续沿用“上传首个 APK 自动建档，不先手工建空记录”的规则；已发布记录若要替换现有二进制，也都必须先下线，再显式替换，再恢复发布，以避免“同一发布记录”在家庭视图中静默漂移成另一个文件。

路由命名也跟随“客户端类型分轨”一起规范化。家庭轻量下载页不再继续堆叠 `/tv-app`、`/phone-app` 这类随手命名，而是朝 `/downloads/<client_type>` 收口，例如 `/downloads/android-tv` 与 `/downloads/android-phone`，让 URL 本身就反映领域模型里的客户端类型。这样后续如果再增加新的客户端轨，也是在同一命名体系内扩展，而不是继续补丁式加新入口。

## 考虑过的替代方案

- **手机端与 TV 端完全拆成两套独立分发系统**：优点是每边都能按自己习惯单独生长，缺点是上传、发布、回退、审计、家庭下载这些明明相同的语义会被复制两遍，管理端页面、后端 API、表结构和测试都会平行膨胀。当前阶段共享部分远多于分叉部分，不值得从第一天就拆成两套系统。
- **手机端与 TV 端共用一个家庭轻量下载页**：看似少一个页面，实际会把不同客户端的下载决策混在同一列表里；TV 端需要 ABI 选择，手机端不需要，终端用户很容易下错客户端安装包。家庭侧分开页比管理侧拆页面更有价值。
- **手机端也沿用 TV 的 ABI 子项模型**：当前 `android-app` 工程没有 TV 那种按 ABI 拆包的事实基础，强行引入“已上传/缺失 ABI”只会让手机端模型过度复杂，并制造并不存在的完整性分支。手机端首期应保留单包发布记录。
- **继续使用随手命名的下载路由，如 `/tv-app` + `/phone-app`**：短期改动最小，但会把“客户端类型分轨”这个上位结构藏掉；一旦后续再扩客户端类型，路由会快速变得零散。按 `/downloads/<client_type>` 规范化命名更能长期承载这套模型。
